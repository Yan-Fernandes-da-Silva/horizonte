# 012 — June 2026 — Phase 09: Career Plan

> Recorded during Phase 09. The roadmap is generated by the Anthropic Claude API,
> with a no-AI fallback. Built per the claude-api skill (official SDK, structured output).

---

## Decisions

1. **Model = `claude-haiku-4-5`** (Yan's choice). The Anthropic guidance defaults to
   `claude-opus-4-8`, but Yan picked Haiku 4.5 for cost — it's the cheapest and is more
   than enough for this structured roadmap generation. The model id lives in
   `src/lib/ai/generate-roadmap.ts` (one constant, easy to change).
2. **Structured output via raw JSON Schema** (`output_config.format = json_schema`),
   not the `zodOutputFormat` helper — the project is on Zod v4 and the raw schema avoids
   any helper/version mismatch. The single text block is `JSON.parse`d and checked
   defensively. (Note: Haiku 4.5 doesn't support the `effort` parameter or adaptive
   thinking, so neither is sent.)
3. **Official SDK, server-only.** `@anthropic-ai/sdk` is called only from the API route
   (`/api/career-plan/create`) so the key never reaches the client. `new Anthropic()`
   reads `ANTHROPIC_API_KEY` from the environment.
4. **No-AI fallback.** If the API fails/refuses, `/career-plan/create` returns `502` and
   the questionnaire shows an error screen with **"Tentar novamente"** and **"Gerar plano
   básico"** (a deterministic `buildFallbackRoadmap` template tailored from the answers).
   This is also how the feature was verified without a live API key.
5. **Context enrichment.** The prompt is enriched with the user's dominant RIASEC types
   (latest completed vocational session) and their most-recent favorited occupation
   (used as the target occupation + stored as `CareerPlan.occupationCode`).

## Persistence & data flow

- `POST /api/career-plan/create` → saves `questionnaire` (answers JSON), generates the
  roadmap, saves `roadmap` (JSON), and creates `CareerTask` rows from short/medium/long
  term (category + sortOrder + status `pending`).
- Tasks are edited via `POST/PUT/DELETE /api/career-plan/[planId]/task[/taskId]`; each
  route verifies the task's plan belongs to the user.
- The roadmap page is a server component; task mutations call the API then `router.refresh()`.

## Scope

- **Implemented:** 6-step questionnaire with progress + loading screen; AI generation +
  fallback; roadmap page with starting/destination, progress bar, **achievement badges**
  (Primeiros passos / Meio caminho / Destino alcançado), task toggle / inline edit /
  delete (with confirm dialog) / add; learning-trail tabs; strategic recommendations;
  future-career badges; plan list/entry page.
- **Deferred** (not in acceptance criteria): drag-and-drop task reordering and the
  "streak" counter — both add dependencies/complexity for little value now.
